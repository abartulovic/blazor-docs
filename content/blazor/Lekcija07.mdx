## Zašto sada uvodimo pretragu, filtriranje i sortiranje?

Kad tablica ima dva ili tri zapisa, gotovo sve je pregledno i bez dodatne logike. Kad broj zapisa naraste, korisnik više ne želi ručno pregledavati red po red. Zato uvodimo tri različita mehanizma:

- **pretragu** – da se brzo pronađu zapisi koji sadrže određeni pojam
- **filtriranje** – da se prikaže samo dio zapisa koji zadovoljava neki uvjet
- **sortiranje** – da se isti zapisi poslože preglednijim redoslijedom

<ConceptGrid>
  <ConceptCard title="Pretraga" icon="search">

  </ConceptCard>
  <ConceptCard title="Filtriranje" icon="filter_alt">

  <ConceptCard title="Sortiranje" icon="swap_vert">
    Svi podaci ostaju vidljivi, ali mijenjamo njihov redoslijed, npr. po imenu ili datumu dolaska.
  </ConceptCard>
</ConceptGrid>

## Korak 3 — Funkcija koja odlučuje prikazuje li se redak u tablici

Sad trebamo napisati metodu koja za svaki zapis odlučuje ostaje li vidljiv ili ne.

~~~csharp
private bool FilterAnimals(AnimalDto animal)
{
    if (!string.IsNullOrWhiteSpace(searchString))
    {
        var term = searchString.Trim().ToLower();

        var matchesSearch =
            (!string.IsNullOrWhiteSpace(animal.Name) && animal.Name.ToLower().Contains(term)) ||
            (!string.IsNullOrWhiteSpace(animal.Breed) && animal.Breed.ToLower().Contains(term)) ||
            (!string.IsNullOrWhiteSpace(animal.Description) && animal.Description.ToLower().Contains(term));

        if (!matchesSearch)
            return false;
    }

    if (!string.IsNullOrWhiteSpace(selectedSpecies))
    {
        if (!string.Equals(animal.Species, selectedSpecies, StringComparison.OrdinalIgnoreCase))
            return false;
    }

    if (adoptedFilter.HasValue)
    {
        if (animal.IsAdopted != adoptedFilter.Value)
            return false;
    }

    return true;
}
~~~

### Kako ova metoda radi?

Metoda prima jedan objekt <code>AnimalDto</code> i na kraju vraća:

- <code>true</code> ako zapis treba ostati prikazan
- <code>false</code> ako ga treba sakriti

Redoslijed je ovakav:

1. prvo se provjerava tražilica
2. zatim se provjerava filter po vrsti
3. zatim se provjerava filter po statusu udomljenja
4. tek ako zapis prođe sve uvjete, ostaje u tablici

Ovo je čitljivije nego jedan golemi uvjet u jednoj liniji.

### Što se ovdje događa?

- stranica šalje HTTP GET zahtjev na <code>api/animals</code>

To je važno: **ne filtriramo bazu direktno iz App sloja**, nego radimo nad listom koju smo već dohvatili preko API-ja.

## Korak 8 — Brisanje zapisa iz iste tablice

Pošto je Lekcija 7 nastavak Lekcije 6, gumb za brisanje može ostati na istoj stranici. Zato treba zadržati i metodu za DELETE zahtjev.

~~~csharp
private async Task DeleteAnimal(int id)
{
    var response = await Http.DeleteAsync($"api/animals/{id}");

    if (response.IsSuccessStatusCode)
    {
        animals = animals.Where(a => a.Id != id).ToList();
    }
}
~~~

### Zašto je ovo dobro rješenje?

- šalje se DELETE zahtjev prema API-ju
- ako API potvrdi uspjeh, odmah uklanjamo zapis iz liste u memoriji
- zato nije potrebno ponovno učitavati cijelu stranicu da bi tablica bila ažurna

## Domaći zadatak

Nastavi nadogradnju modula <code>HousingUnits</code> koji se provlači iz prethodnih lekcija i primijeni isti obrazac kao na stranici <code>Animals</code>.

### Potrebno je napraviti:

1. Stranicu s popisom smještajnih jedinica nadogradi tako da koristi:
   - tekstualnu pretragu
   - barem jedan padajući filter
   - sortiranje po barem tri stupca
2. Dodaj gumb za resetiranje svih filtera.
3. Dodaj poruku koja se prikazuje kad nema rezultata.
4. Prikaži prazne ili nullable vrijednosti sigurno, bez rušenja stranice.
5. Ako postoji gumb za brisanje, nakon uspješnog DELETE zahtjeva odmah ažuriraj listu bez ručnog refreshanja stranice.

### Obavezni kriteriji završnog rada

Na kraju domaćeg rješenje mora:

- dohvaćati podatke preko HTTP + DTO pristupa
- imati preglednu traku za upravljanje tablicom iznad popisa
- imati jasno odvojenu logiku za pretragu i filtriranje
- imati barem jednu sortirajuću tablicu s više stupaca
- koristiti isti stil i pristup kao stranica <code>Animals</code> iz ove lekcije

## Kratak sažetak

U ovoj lekciji nismo mijenjali način komunikacije s bazom, nego smo nadogradili korisnički prikaz podataka koje već dohvaćamo iz API-ja.

Tijek rada sada izgleda ovako:

1. Stranica <code>Animals.razor</code> preko <code>HttpClient</code>-a šalje GET zahtjev na <code>api/animals</code>.
2. API kontroler vraća listu <code>AnimalDto</code> objekata.
3. Ta lista se sprema u <code>animals</code> i prikazuje u <code>MudTable</code> tablici.
4. Korisnik iznad tablice upisuje pojam za pretragu ili odabire filtere.
5. <code>FilterAnimals</code> za svaki zapis provjerava treba li ostati vidljiv.
6. <code>MudTableSortLabel</code> omogućuje dodatno sortiranje bez mijenjanja samih podataka.
7. Ako korisnik obriše zapis, šalje se DELETE zahtjev i lista se odmah ažurira u memoriji.

To znači da je ova lekcija prirodan nastavak Lekcije 6: tamo smo izgradili CRUD, a ovdje smo postojeći popis pretvorili u stvarno upotrebljiv radni pregled.
