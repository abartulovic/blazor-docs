## 4. Model `Animal`

Ako si u prethodnoj lekciji već izradio model, sada ga samo provjeri i po potrebi uskladi. Za ovu lekciju dovoljan je ovakav model:

```csharp
namespace AzilEdu.Shared.Models;

public class Animal
{
    public int Id { get; set; }
    public string Name { get; set; } = string.Empty;
    public string Species { get; set; } = string.Empty;
    public string Breed { get; set; } = string.Empty;
    public int Age { get; set; }
    public DateTime ArrivalDate { get; set; }
    public bool IsAdopted { get; set; }
}
```

<InfoBox type="info" title="Zašto nam je ovaj model dovoljan za sada?">
Ne trebamo odmah ubacivati deset dodatnih polja. Učeniku je lakše razumjeti CRUD kada radi nad manjim i preglednim modelom. Kasnije se model može proširivati novim poljima i relacijama.
</InfoBox>

## 7. Zašto odmah prikazujemo i gumbe Uredi i Obriši?

Zato što želimo da učenici odmah vide puni smisao tabličnog prikaza. Ako imamo samo listu bez akcija, stranica izgleda statično. Kada uz svaki red dodamo mogućnost uređivanja i brisanja, učenici lakše povezuju pojam:

- jedan red tablice = jedan zapis u bazi
- akcija nad redom = akcija nad tim zapisom

To je vrlo važna mentalna veza.

## 11. Brisanje zapisa

Najjednostavnija verzija brisanja već je prikazana na listi. Ipak, učenicima treba jasno objasniti što se događa.

```csharp
private async Task DeleteAnimal(int id)
{
    var animal = await Db.Animals.FindAsync(id);

    if (animal is not null)
    {
        Db.Animals.Remove(animal);
        await Db.SaveChangesAsync();
        animals = await Db.Animals.ToListAsync();
    }
}
```

### Objašnjenje korak po korak

1. pronađemo zapis koji želimo obrisati
2. ako zapis postoji, proslijedimo ga metodi <code>Remove()</code>
3. pozivom <code>SaveChangesAsync()</code> promjena se stvarno upisuje u bazu
4. nakon toga ponovno učitamo listu kako bi korisnik odmah vidio osvježeno stanje

<InfoBox type="warning" title="Je li dovoljno odmah obrisati zapis?">
Tehnički jest, ali za stvarne aplikacije je bolje dodati <strong>potvrdu brisanja</strong>. Tako korisnik ne može slučajno kliknuti gumb i izgubiti podatke bez upozorenja.
</InfoBox>

## 13. Osvježavanje prikaza nakon unosa, izmjene i brisanja

Vrlo je važno da učenici razumiju zašto se nakon CRUD operacije korisnik mora vratiti na osvježenu listu.

Ako to ne napravimo:

- novi zapis možda neće biti vidljiv odmah
- obrisani zapis može ostati prikazan dok se stranica ručno ne osvježi
- korisnik će misliti da aplikacija ne radi ispravno

Zato je potpuno opravdano da nakon promjene:

- ili ponovno dohvatimo podatke iz baze
- ili napravimo navigaciju natrag na listu koja se ponovno inicijalizira

## 14. Česta pitanja učenika

### "Zašto ne mijenjamo samo listu u memoriji?"

Zato što želimo raditi nad bazom. Lista u memoriji nestaje kad se aplikacija ugasi ili osvježi. Baza služi upravo zato da podaci ostanu sačuvani.

### "Zašto kod uređivanja ne zovemo `Db.Animals.Update(animal)`?"

Može se i to koristiti, ali u ovom slučaju objekt je već učitan iz baze preko `DbContexta`, pa EF Core već prati njegove promjene. Zato je dovoljno promijeniti vrijednosti i pozvati `SaveChangesAsync()`.

### "Zašto nakon brisanja opet dohvaćamo listu?"

Zato da korisnik odmah vidi trenutno stanje iz baze. Inače prikaz može ostati zastario.

Za domaći učenici trebaju proširiti funkcionalnost na barem jedan od ova tri načina:

1. dodati validaciju da ime i vrsta ne mogu ostati prazni
2. dodati potvrdu prije brisanja
3. dodati još jedno polje u model, npr. <code>Color</code> ili <code>Description</code>, pa zatim uskladiti prikaz i formu

<InfoBox type="warning" title="Ako učenik dodaje novo polje u model">
Treba ga podsjetiti da nakon promjene modela možda mora napraviti novu migraciju i ažurirati bazu, inače će model i baza biti neusklađeni.
</InfoBox>

## 19. Što slijedi dalje?

Nakon ove lekcije projekt već ima pravi osjećaj aplikacije:

- postoji baza
- postoje stvarni podaci
- korisnik može dodavati, uređivati i brisati zapise

Sljedeći korak može biti jedan od ova dva smjera:

- poboljšanje UX-a: validacija, dialog potvrde, bolji prikaz statusa i detalja
- arhitekturno poboljšanje: odvajanje logike u servise, API sloj i DTO pristup

Za učenike početnike obično je pametnije prvo još malo učvrstiti CRUD, a tek onda ići na složeniju arhitekturu.
