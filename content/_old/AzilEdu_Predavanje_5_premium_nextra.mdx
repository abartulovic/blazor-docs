## Gdje smo stali?

U prethodnoj lekciji:

- pripremili smo bazu
- dodali **Entity Framework Core**
- izradili `AzilEduDbContext`
- postavili `DbSet<Animal>`
- definirali connection string
- napravili migraciju i stvorili bazu

To znači da naš projekt više nije samo vizualni prikaz i rad s probnim podacima. Sada imamo temelje za **pravu pohranu podataka**.

## Što radimo danas?

U ovoj lekciji izrađujemo **prvu stvarnu funkcionalnost aplikacije**:

1. dodajemo stranicu za unos nove životinje
2. izrađujemo formu s MudBlazor komponentama
3. povezujemo polja forme s modelom
4. ubacujemo `DbContext`
5. spremamo novu životinju u bazu
6. prikazujemo poruku o uspjehu ili grešci
7. po potrebi preusmjeravamo korisnika na popis životinja

## Zašto ne unosimo podatke direktno u tablicu?

Početnici često pomisle da je aplikacija tu samo da "upiše nešto u bazu". To je prejednostavan pogled.

U stvarnosti aplikacija mora:

- prikazati korisniku razumljivu formu
- ograničiti i usmjeriti unos
- provjeriti ispravnost podataka
- spriječiti spremanje besmislenih vrijednosti
- spremiti podatke na kontroliran način

Zato radimo formu u aplikaciji, a ne ručni unos direktno u bazu.

<InfoBox type="info" title="Pedagoška napomena">
Učenicima treba naglasiti da forma nije "ukras". Ona je kontrolirani ulaz u sustav. Ako je forma loše napravljena, korisnici unose loše podatke, a loši podaci kasnije kvare cijelu aplikaciju.
</InfoBox>

## 1. Priprema modela `Animal`

Prije izrade forme provjerimo model. Ako u prethodnoj lekciji još nismo fino doradili svojstva, sada je pravi trenutak.

Primjer modela:

```csharp
namespace AzilEdu.Shared.Models;

public class Animal
{
    public int Id { get; set; }
    public string Name { get; set; } = string.Empty;
    public string Species { get; set; } = string.Empty;
    public string? Breed { get; set; }
    public int Age { get; set; }
    public DateTime ArrivalDate { get; set; } = DateTime.Today;
    public bool IsAdopted { get; set; }
}
```

### Objašnjenje svojstava

- `Id` — primarni ključ
- `Name` — ime životinje
- `Species` — vrsta, npr. pas ili mačka
- `Breed` — pasmina, nije uvijek obavezna
- `Age` — dob u godinama
- `ArrivalDate` — datum dolaska u azil
- `IsAdopted` — je li životinja već udomljena

<InfoBox type="warning" title="Važno">
Ako sada promijeniš model `Animal`, a baza je već napravljena po staroj verziji modela, moraš napraviti **novu migraciju**. Učenici trebaju shvatiti da model i baza moraju ostati usklađeni.
</InfoBox>

## 4. Priprema objekta za unos

Da bi forma imala kamo zapisivati podatke, trebamo pripremiti objekt.

U stranici `AddAnimal.razor` dodaj:

```razor
@using AzilEdu.Shared.Models

@code {
    private Animal newAnimal = new();
}
```

### Što to znači?

- `newAnimal` je objekt u koji će se upisivati vrijednosti iz forme
- kad korisnik upisuje ime, to ide u `newAnimal.Name`
- kad odabere datum dolaska, to ide u `newAnimal.ArrivalDate`

To je srž bindinga u Blazoru.

<InfoBox type="info" title="Kako ovo objasniti učenicima?">
Najjednostavnije je reći ovako: forma nije zasebna od koda. Svako polje na ekranu povezano je s jednim svojstvom objekta u memoriji. Korisnik zapravo "puni" objekt kroz korisničko sučelje.
</InfoBox>

## 8. Izrada metode `SaveAnimal`

Sada dolazimo do ključnog dijela lekcije: spremanja podataka.

Primjer metode:

```razor
@code {
    private Animal newAnimal = new();

    private async Task SaveAnimal()
    {
        DbContext.Animals.Add(newAnimal);
        await DbContext.SaveChangesAsync();

        newAnimal = new Animal();
    }
}
```

### Što ova metoda radi?

- `DbContext.Animals.Add(newAnimal);`
  - EF Core počinje pratiti novi objekt
- `await DbContext.SaveChangesAsync();`
  - promjene se stvarno upisuju u bazu
- `newAnimal = new Animal();`
  - forma se resetira nakon uspješnog spremanja

## 9. Dodavanje osnovne validacije prije spremanja

Ne želimo spremati životinju bez imena ili vrste. Zato uvodimo provjeru.

Jedna jednostavna verzija može izgledati ovako:

```razor
private async Task SaveAnimal()
{
    if (string.IsNullOrWhiteSpace(newAnimal.Name) || string.IsNullOrWhiteSpace(newAnimal.Species))
    {
        return;
    }

    DbContext.Animals.Add(newAnimal);
    await DbContext.SaveChangesAsync();

    newAnimal = new Animal();
}
```

### Što ovdje provjeravamo?

- je li korisnik upisao ime
- je li korisnik upisao vrstu

Ako nije, metoda jednostavno prekida izvršavanje.

<InfoBox type="warning" title="Ovo još nije idealna validacija">
Korisnik ovdje još ne dobiva poruku zašto spremanje nije uspjelo. Dakle, validacija postoji, ali korisničko iskustvo još nije dovoljno dobro. Zato u sljedećem koraku dodajemo povratnu informaciju.
</InfoBox>

## 11. Dodavanje obrade iznimki

U stvarnoj aplikaciji spremanje može pasti. Na primjer:

- baza nije dostupna
- connection string nije ispravan
- projekt je loše konfiguriran
- dogodila se neočekivana greška

Zato metoda treba biti robusnija.

```razor
private async Task SaveAnimal()
{
    try
    {
        if (string.IsNullOrWhiteSpace(newAnimal.Name) || string.IsNullOrWhiteSpace(newAnimal.Species))
        {
            Snackbar.Add("Ime i vrsta su obavezni.", Severity.Warning);
            return;
        }

        DbContext.Animals.Add(newAnimal);
        await DbContext.SaveChangesAsync();

        Snackbar.Add("Životinja je uspješno spremljena.", Severity.Success);
        newAnimal = new Animal();
    }
    catch (Exception ex)
    {
        Snackbar.Add($"Došlo je do greške: {ex.Message}", Severity.Error);
    }
}
```

<InfoBox type="info" title="Kako ovo objasniti učenicima?">
Ne trebaju učenici sada ulaziti u svaku vrstu iznimke. Dovoljno je da razumiju osnovnu ideju: program ne smije puknuti bez kontrole. Umjesto toga treba uhvatiti grešku i korisniku pokazati smislen odgovor.
</InfoBox>

## 12. Preusmjeravanje nakon uspješnog spremanja

Nakon spremanja možemo korisnika:

- ostaviti na istoj stranici
- očistiti formu
- ili ga preusmjeriti na popis životinja

Često je za ovakav projekt najbolje preusmjeravanje na popis.

Na vrh stranice dodaj:

```razor
@inject NavigationManager Navigation
```

Zatim u metodu:

```razor
Snackbar.Add("Životinja je uspješno spremljena.", Severity.Success);
Navigation.NavigateTo("/animals");
```

### Kada je bolje ostati na istoj stranici?

Ako korisnik treba unijeti više životinja zaredom, onda možda ima smisla:

- prikazati poruku uspjeha
- očistiti formu
- ostaviti korisnika na istoj stranici

Dakle, ovdje nema samo jednog točnog rješenja. Treba objasniti učenicima da dizajn aplikacije ovisi i o načinu korištenja.

@code {
    private Animal newAnimal = new();

    private async Task SaveAnimal()
    {
        try
        {
            if (string.IsNullOrWhiteSpace(newAnimal.Name) || string.IsNullOrWhiteSpace(newAnimal.Species))
            {
                Snackbar.Add("Ime i vrsta su obavezni.", Severity.Warning);
                return;
            }

            DbContext.Animals.Add(newAnimal);
            await DbContext.SaveChangesAsync();

            Snackbar.Add("Životinja je uspješno spremljena.", Severity.Success);
            Navigation.NavigateTo("/animals");
        }
        catch (Exception ex)
        {
            Snackbar.Add($"Došlo je do greške: {ex.Message}", Severity.Error);
        }
    }
}
