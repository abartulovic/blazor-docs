U prethodnoj lekciji modul <code>Animals</code> već radi preko <code>HttpClient</code> poziva, DTO modela i API kontrolera. U ovoj lekciji taj obrazac ne mijenjamo. Mijenjamo samo strukturu baze i zatim redom prilagođavamo sve dijelove projekta da ponovno rade zajedno.
</InfoBox>

## Zašto u ovoj lekciji ne uvodimo sve relacije odjednom

Kad se prvi put uvode relacije u bazu, najčešća pogreška je da se odjednom promijeni previše toga:

- model
- `DbContext`
- seed podaci
- DTO modeli
- kontroleri
- UI stranice
- migracije

Tada je vrlo teško pratiti što se točno promijenilo i zašto aplikacija više ne radi.

Zato ćemo u ovoj lekciji prvo izdvojiti samo **status životinje** u novu tablicu `AnimalStatuses`.  
Sve ostalo iz prethodne lekcije ostaje isto:

- `Species` ostaje tekst
- `Breed` ostaje tekst
- ostatak `Animals` modula radi kao i do sada

Kad na ovom jednom primjeru prođemo cijeli ciklus i aplikacija ponovno proradi, tek tada ćemo pokazati kako se isti obrazac primjenjuje i na ostale veze.

---

## Što je cilj ove lekcije

Na kraju ove lekcije projekt će imati:

- novu tablicu `AnimalStatuses`
- `Animal` više neće spremati status kao običnu `bool` vrijednost
- `Animal` će spremati `AnimalStatusId`
- API će vraćati naziv statusa kroz DTO
- create i edit forme koristit će dropdown listu za status
- seed podaci bit će dodani prije nove migracije
- baza će biti ponovno izgrađena čistom migracijom

## Najvažniji redoslijed rada u ovoj lekciji

Ovdje je redoslijed izuzetno važan.

Ne radi se ovako:

1. malo promijenimo model
2. odmah pokrenemo migraciju
3. kasnije popravljamo kontrolere
4. još kasnije ispravljamo UI

To gotovo uvijek završi build greškama i neuspjelom migracijom.

U ovoj lekciji radi se ovako:

1. napravi novu tablicu `AnimalStatus`

10. provjeri da projekt prolazi build

<InfoBox type="warning" title="Migracija se radi tek kad build prolazi">
Ako projekt ne prolazi build, migracija neće uspjeti. Zato se baza i migracije ne diraju dok svi modeli, DTO-i, kontroleri i UI stranice ne budu međusobno usklađeni.
</InfoBox>

<InfoBox type="warning" title="Stara baza sada više nije ispravna">
Prethodna baza sadrži stare zapise i staru strukturu iz vremena kada je status bio spremljen kao <code>bool IsAdopted</code>. Nakon uvođenja tablice <code>AnimalStatuses</code> ta baza više nije dobra osnova za nastavak rada. Zato je nakon uspješnog builda potrebno izbrisati staru <code>.db</code> bazu i stare migracije, pa tek onda napraviti novu čistu migraciju.
</InfoBox>

## Korak 1 – napravi tablicu `AnimalStatus`

Počinjemo od nove lookup tablice koja će sadržavati moguće statuse životinje.

### Datoteka: `AzilEdu.Shared/Models/AnimalStatus.cs`

~~~csharp
namespace AzilEdu.Shared.Models;

public class AnimalStatus
{
    public int Id { get; set; }
    public string Name { get; set; } = string.Empty;

    public ICollection<Animal> Animals { get; set; } = new List<Animal>();
}
~~~

### Što ova tablica predstavlja

Umjesto da status spremamo kao `true/false`, sada ga spremamo kao zapis u zasebnoj tablici.

Primjeri statusa:

- Dostupna za udomljenje
- Rezervirana
- Udomljena
- Na liječenju

Taj pristup je puno fleksibilniji jer kasnije možeš dodavati nova stanja bez promjene strukture `Animals` tablice.

### Što znači ova linija u modelu

~~~csharp
public ICollection<Animal> Animals { get; set; } = new List<Animal>();
~~~

Ova linija znači da jedan zapis u tablici `AnimalStatuses` može biti povezan s više životinja.

Primjer:
- status **Dostupna za udomljenje** može imati više povezanih životinja
- status **Na liječenju** isto tako može imati više povezanih životinja

`ICollection<Animal>` predstavlja kolekciju svih životinja koje imaju taj status.  
`new List<Animal>()` odmah inicijalizira praznu listu kako svojstvo ne bi bilo `null`.

To svojstvo se naziva **navigacijsko svojstvo** i Entity Framework ga koristi kako bi razumio odnos:

- jedan `AnimalStatus`
- više `Animal` zapisa

## Korak 3 – proširi `AzilEduDbContext`

Sada `DbContext` mora znati da postoji nova tablica i nova relacija.

### Datoteka: `AzilEdu.Api/Data/AzilEduDbContext.cs`

Prvo dodaj novi `DbSet`:

~~~csharp
public DbSet<Animal> Animals => Set<Animal>();
public DbSet<AnimalStatus> AnimalStatuses => Set<AnimalStatus>();
~~~

Zatim u metodi `OnModelCreating` dodaj relaciju:

~~~csharp
modelBuilder.Entity<Animal>()
    .HasOne(a => a.AnimalStatus)
    .WithMany(s => s.Animals)
    .HasForeignKey(a => a.AnimalStatusId)
    .OnDelete(DeleteBehavior.Restrict);
~~~

### Gdje se nalazi `OnModelCreating`

`OnModelCreating` se nalazi unutar klase `AzilEduDbContext` u datoteci:

`AzilEdu.Api/Data/AzilEduDbContext.cs`

To je mjesto na kojem Entity Framework dodatno konfigurira odnose između tablica i početne seed podatke.

## Korak 6 – prepravi `AnimalsController`

Kontroler se mora prilagoditi novoj relaciji. Sada status više ne dolazi iz `IsAdopted`, nego iz povezane tablice `AnimalStatuses`.

### Datoteka: `AzilEdu.Api/Controllers/AnimalsController.cs`

### Dohvat svih životinja

~~~csharp
[HttpGet]
public async Task<ActionResult<List<AnimalDto>>> GetAnimals()
{
    var animals = await _context.Animals
        .Include(a => a.AnimalStatus)
        .OrderBy(a => a.Name)
        .Select(a => new AnimalDto
        {
            Id = a.Id,
            Name = a.Name,
            Species = a.Species,
            Breed = a.Breed,
            Gender = a.Gender,
            Age = a.Age,
            ArrivalDate = a.ArrivalDate,
            AnimalStatusId = a.AnimalStatusId,
            Status = a.AnimalStatus != null ? a.AnimalStatus.Name : string.Empty,
            ImageUrl = a.ImageUrl,
            Description = a.Description
        })
        .ToListAsync();

    return Ok(animals);
}
~~~

### Što je ovdje važno

- `Include(a => a.AnimalStatus)` učitava povezanu tablicu
- `Status = a.AnimalStatus.Name` vraća naziv statusa za prikaz
- `AnimalStatusId` ostaje dostupan za edit formu

### Dodavanje nove životinje

~~~csharp
[HttpPost]
public async Task<ActionResult<AnimalDto>> CreateAnimal(SaveAnimalDto dto)
{
    var animal = new Animal
    {
        Name = dto.Name,
        Species = dto.Species,
        Breed = dto.Breed,
        Gender = dto.Gender,
        Age = dto.Age,
        ArrivalDate = dto.ArrivalDate,
        AnimalStatusId = dto.AnimalStatusId,
        ImageUrl = dto.ImageUrl,
        Description = dto.Description
    };

    _context.Animals.Add(animal);
    await _context.SaveChangesAsync();

    var createdAnimal = await _context.Animals
        .Include(a => a.AnimalStatus)
        .FirstAsync(a => a.Id == animal.Id);

    var result = new AnimalDto
    {
        Id = createdAnimal.Id,
        Name = createdAnimal.Name,
        Species = createdAnimal.Species,
        Breed = createdAnimal.Breed,
        Gender = createdAnimal.Gender,
        Age = createdAnimal.Age,
        ArrivalDate = createdAnimal.ArrivalDate,
        AnimalStatusId = createdAnimal.AnimalStatusId,
        Status = createdAnimal.AnimalStatus != null ? createdAnimal.AnimalStatus.Name : string.Empty,
        ImageUrl = createdAnimal.ImageUrl,
        Description = createdAnimal.Description
    };

    return CreatedAtAction(nameof(GetAnimalById), new { id = createdAnimal.Id }, result);
}
~~~

Na isti način treba prilagoditi i `GetAnimalById` te `UpdateAnimal`.

## Korak 7 – napravi lookup kontroler za statuse

Dropdown u formama treba endpoint koji vraća sve moguće statuse.

### Datoteka: `AzilEdu.Api/Controllers/AnimalStatusesController.cs`

~~~csharp
using AzilEdu.Api.Data;
using AzilEdu.Shared.DTOs;
using Microsoft.AspNetCore.Mvc;
using Microsoft.EntityFrameworkCore;

namespace AzilEdu.Api.Controllers;

[ApiController]
[Route("api/[controller]")]
public class AnimalStatusesController : ControllerBase
{
    private readonly AzilEduDbContext _context;

    public AnimalStatusesController(AzilEduDbContext context)
    {
        _context = context;
    }

    [HttpGet]
    public async Task<ActionResult<List<LookupDto>>> GetAnimalStatuses()
    {
        var result = await _context.AnimalStatuses
            .OrderBy(x => x.Name)
            .Select(x => new LookupDto
            {
                Id = x.Id,
                Name = x.Name
            })
            .ToListAsync();

        return Ok(result);
    }
}
~~~

### Zašto je ovaj kontroler potreban

Bez ovog endpointa create i edit forma neće moći dohvatiti statuse.

if (!string.IsNullOrWhiteSpace(selectedStatus) &&

## Korak 12 – izbriši stare migracije i staru bazu

Pošto se model u ovoj lekciji ozbiljnije mijenja, najčišći pristup je:

1. izbriši postojeću `.db` datoteku

3. još jednom provjeri build

### Zašto je to sada dobro rješenje

Ako su prethodne migracije nastajale dok se model više puta mijenjao, vrlo lako ostane nered.  
Zato je u ovoj fazi urednije krenuti od čiste baze i jedne nove migracije koja točno prati konačno stanje modela.

## Domaći zadatak

Domaći se nastavlja na postojeći modul `HousingUnits` i sada je cilj primijeniti isti obrazac koji je u ovoj lekciji prošao `AnimalStatus`.

### Što treba napraviti

1. Odaberi jedno svojstvo iz `HousingUnits` koje je trenutno spremljeno kao običan tekst ili `bool`
2. Napravi novu lookup tablicu za to svojstvo
3. U glavnom modelu dodaj strani ključ i navigacijsko svojstvo
4. Proširi `DbContext`
5. U `OnModelCreating` dodaj relaciju i seed podatke
6. Ako postoji seed za glavnu tablicu, prilagodi ga da koristi novi strani ključ
7. Prepravi DTO modele tako da imaju:
   - ID povezane tablice
   - tekstualni naziv za prikaz
8. Napravi lookup kontroler koji vraća listu za dropdown
9. Prepravi create formu tako da koristi dropdown umjesto starog tekstualnog unosa
10. Prepravi edit formu na isti način
11. Prepravi listu tako da prikazuje naziv povezanog zapisa
12. Provjeri da projekt prolazi build
13. Tek nakon toga napravi migraciju

### Što treba predati

Domaći je gotov kad pokažeš:

- novi model lookup tablice
- izmijenjeni glavni model
- izmijenjeni `DbContext`
- seed podatke
- lookup kontroler
- create formu
- edit formu
- listu s prikazom naziva iz povezane tablice
- uspješan build
- uspješnu migraciju i ažuriranu bazu

### Na što posebno pripaziti

- ne radi migraciju prije uspješnog builda
- ne ostavljaj stara polja i nova polja zajedno ako je stari pristup zamijenjen novim
- DTO, kontroler i UI moraju biti usklađeni s modelom baze
- lookup endpoint mora postojati prije nego što create/edit forme pokušaju dohvatiti podatke

## Kratak sažetak

U ovoj lekciji `Animals` modul nije odjednom prebačen na više relacija, nego je prvo na jednom jasnom primjeru uvedena tablica `AnimalStatuses`. Najprije je napravljen novi model, zatim je prilagođen `Animal`, proširen `DbContext`, dodani su seed podaci, usklađeni DTO modeli i API kontroleri, pa su tek onda prepravljene UI stranice. Nakon uspješnog builda izbrisane su stare migracije i stara baza te je napravljena nova migracija. Na kraju je isti obrazac detaljno prikazan i za buduće veze prema `Species` i `Breed`.
